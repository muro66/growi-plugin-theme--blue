# 問関構（Question → Relation → Structure）実装計画（v0〜v2）

## 0. 前提（確定していること）
- 課題は「工程管理ツール不足」ではなく、**認知と行動の流れが設計されていない**こと。
- 真の課題は **「見なくても成立してしまう構造」** 。
- 解くべきは「人が自分の立ち位置を見ない」問題であり、依頼・監視でなく **構造で“見ざるを得ない”状態に誘導**する。
- LLMは効率化のためではなく、**思考を生む問い生成装置**として扱う（回答者ではない）。

---

## 1. 成果物の定義（何を作れば“使える”になるか）

### 1.1 スキーマ（Structure）を固定する
最低限、以下の3要素だけを扱う（最小で動かす）。

- Question（問い）: 何を考えさせるか（テンプレ）
- Intent（意思表明）: 人が埋めるべき結論（必須項目を最小にする）
- Relations（関係）: 何と何がつながっているか（影響のリンク）

GROWI側では次のような“保存単位”を作る（例）。

```markdown
```ticket-intent
question: （LLMまたはテンプレで生成）
intent: （人が埋める。次アクション/判断/条件）
relations:
  - type: blocker|dependency|risk|impact
    target: /projects/<name>/tasks/<task>
    note: （根拠や短いメモ）
```
```

※あなたのネーミング `TAG5 / FLOW-RING / スキーマ` は、この保存単位へのマッピングに落とし込む。

### 1.2 GROWIページ構造（どこに何を書くか）

- プロジェクトページ `/projects/<name>`
  - `## Tasks` : `$lsx(/projects/<name>/tasks, depth=1, sort=path)`
  - `## Watch / Questions` : 未意思表明（未入力）タスクを強調表示

- タスクページ `/projects/<name>/tasks/<task>`
  - 既存: `ticket-meta`（status / project / assignee / startDate / dueDate / progress）
  - 追加: `ticket-intent`（問い＋人の意思表明＋関係）

---

## 2. 解決のコア：関門化（見ないと進めない）

“見てください”では弱いので、**状態変更・更新に関門を付ける**。

### 関門トリガー1：状態変更（強）
- 例: status / progress / dueDate 等を更新する前に、
  - `intent` の必須入力が無い場合は保存不可
  - もしくは保存はできても「未意思表明」フラグを付与

→ **「閲覧」→「意思表示」**に昇格させる。

### 関門トリガー2：重要ビュー（見ざるを得ない）
- ダッシュボードで
  - 未意思表明タスクだけハイライト
  - クリックで「問い→入力フォーム」へ誘導

→ “見る”が目的ではなく、**“埋める”に接続**する。

---

## 3. 接続（Relation）の自動化ルール

- relations は最小で良い
  - 影響先タスク（リンク）
  - 種別（blocker / dependency / risk / impact）
  - 根拠メモ（短くて良い）

- 推定は自動化してよいが、誤りの許容度を下げるために
  - 最終確定は人に寄せる
  - まずは「次アクション」「判断理由」「影響（だれ/何へ）」のみに絞る

---

## 4. LLMの役割転換（重要）

### 旧来：人が質問 → AIが回答
### 今回：AIが問い → 人が思考

LLMは以下だけを行う：
- 関門が発火したタイミングで、タスク本文と `ticket-meta` を見て **問いを1つ生成**
- question テンプレは強制（スキーマ準拠）

“回答を埋める”のは人にする。これが思考の発生装置になる。

---

## 5. 実装ロードマップ（v0〜v2）

### v0（最短）：見ざるを得ない表示だけ作る
- プロジェクトダッシュボード
  - status集計、期限ハイライト、タスク一覧
  - `ticket-intent` が未入力のタスクを見える化
- クリックでフォームへ誘導（フォームは固定テンプレでOK）
- LLM無しでも成立させる（テンプレ問い）

### v1（関門化）：保存を必須入力へ接続する
- チケット更新UI（status/progress/dueDate更新等）に関門を入れる
- 必須入力が空なら
  - 保存不可、または未意思表明フラグの付与

### v2（LLM追加）：問い生成を自動化する
- 関門発火時にだけ LLM を呼び
  - question を生成してフォームへ差し込む
- ここでも intent の最終埋めは人にする

---

## 6. 検証（ツールではなく流れが変わったか）

次で評価する：
- 未意思表明タスク率が減るか
- 状態変更に付随する意思表明率が上がるか
- “会話/依頼の停滞”が減るか（運用ログで観測）

---

## 7. 未確定事項（次に詰めたい1点）
あなたの `TAG5 / FLOW-RING` の意味を、このスキーマにどう割り当てるかを確定したい。

- TAG5 は「意思表明に必須の5項目」を意味しているか？
- FLOW-RING は「問い→入力→リンク→再利用」の順序制御を意味しているか？

この2点が確定すれば、次は
- GROWI用ページ雛形（コピペ用）
- ダッシュボード UI の未入力強調ロジック
- 状態更新UIの関門実装仕様（入力必須チェック）
まで一気に落とせます。
