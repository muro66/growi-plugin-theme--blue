# 個別に作成する vs GROWI スクリプトプラグイン — どちらが実装しやすいか

結論から言うと、**「GROWI のデータだけ使う・GROWI の画面で完結する機能」はスクリプトプラグインの方が実装しやすい**ことが多い。**独自の永続データや重い処理をサーバーでやりたい場合は個別アプリの方が向く。**

---

## スクリプトプラグインの方が向いているケース（実装しやすい）

| 条件 | 理由 |
|------|------|
| **表示するデータがすべて GROWI にある** | 認証・データ取得を GROWI API に任せられる。別バックエンド不要。 |
| **「今開いているページ」や「ツリー・リンク」が主役** | `growiFacade` や GROWI API でそのまま扱える。 |
| **UI が「GROWI のページ／サイドバーの延長」でよい** | 既存レイアウトに乗せるだけでよく、ゼロから画面を作らない。 |
| **状態をあまり持たない（または localStorage で足りる）** | サーバー側の DB が不要。 |

**例（あなたのプロジェクトから）**

- **リンクのグラフ表示** … ページ・リンクは GROWI API で取得 → フロントでグラフ描画だけ。個別アプリだと「GROWI 連携・認証・同期」を自分で作る必要がある。
- **BOM / FMEA 風テーブル表示** … 本文 Markdown をパースして表示するだけ。データ保存は GROWI の編集に任せられる。
- **2D マップ・本棚ビュー** … ツリー／子ページ一覧を API で取って表示するだけ。
- **見出しの構造抽出・Knowledge Schema（YAML）表示** … クライアントで Markdown/YAML を読むだけ。

→ これらは **スクリプトプラグインの方が実装しやすい**。

---

## 個別アプリの方が向いているケース

| 条件 | 理由 |
|------|------|
| **GROWI 以外の DB やサービスを常時使う** | 例: 専用のグラフ DB、CE 用の nodes/edges ストア（WEB_Tool のような）は別サーバーが自然。 |
| **重い処理をサーバーでやりたい** | OCR（NDLOCR-Lite）、大規模グラフ計算、バッチなどはブラウザでは無理。 |
| **GROWI と独立した「ツール」として使いたい** | デスクトップアプリ（Dev_tool の Tauri）や、別ドメインの Web アプリのまま運用したい場合。 |
| **複数ツールの統合ポータルにしたい** | Knowledge_Library のように「図書館＋検索＋3D＋グラフ」を一つのアプリとして提供したい場合は、個別アプリのままの方が設計しやすい。 |

**例**

- **WEB_Tool の CE 統合フレーム全体** … 独自 DB（nodes/edges）、import_md、3D など → 個別アプリのままがよい。必要なら「GROWI のページをノードとして取り込む」連携を API で追加。
- **NDLOCR-Lite の OCR エンジン** … Python + ONNX → 個別アプリ/サービス。GROWI プラグインでは「OCR 結果を貼り付ける UI」程度が現実的。
- **Knowledge_Library を「図書館ポータル」としてそのまま提供** … 複数画面・バックエンド・GROWI 連携を一括で持つなら、現状の個別アプリの方が管理しやすい。

---

## ハイブリッド（組み合わせ）

- **メインは個別アプリ**で、**GROWI 上での「入り口」や「プレビュー」だけスクリプトプラグイン**にする、という形もあり。
  - 例: グラフ計算は別サーバー、GROWI 上では「このページを中心にしたグラフを表示」ボタンでそのサーバーを呼ぶプラグイン。
- **データは GROWI のみ**だが、**表示が重い（3D など）**場合は、まずスクリプトプラグインで 2D 版を作り、必要なら 3D は別アプリ/別タブで提供する、という段階もあり。

---

## まとめ

| やりたいこと | おすすめ |
|--------------|----------|
| GROWI のページ・リンク・ツリーの「見せ方」を変える（グラフ、マップ、本棚、BOM/FMEA 表示など） | **スクリプトプラグイン**の方が実装しやすい |
| GROWI 以外の DB や重い処理（OCR、CE 専用ストアなど）が中心 | **個別アプリ**の方がよい。必要に応じてプラグインで「GROWI から呼び出す窓」を用意 |
| 「図書館ポータル」のように複数画面・複数機能を一つのアプリとして提供したい | **個別アプリ**を主に。GROWI 連携は API で |

**あなたのリストの多く（グラフ表示、BOM/FMEA 表示、2D マップ、本棚、見出し構造、Schema 表示）は、GROWI のデータと画面の上で完結するので、個別にアプリを作るより、GROWI のスクリプトプラグインとして実装した方が実装しやすい。**
